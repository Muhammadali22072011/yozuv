### 1. [HIGH/bug] X-ing out of the dashboard tour force-starts the full 7-page onboarding chain
- **Where:** `frontend/src/app/dashboard/page.tsx:161-171, 497-501`
- **Problem:** The dashboard's own Tour is rendered with onClose={finishDashboardTour} (line 500), and finishDashboardTour() unconditionally calls startOnboarding() after marking the tour seen. Tour.tsx fires onClose both when the user finishes (the 'Tayyor' button on the last info step, Tour.tsx:216) AND when the user clicks the X / 'O'tkazib yuborish' skip button (Tour.tsx:198, 208). So a user who deliberately aborts the dashboard tour mid-way is still dropped into the entire 7-page guided onboarding sequence and navigated away to /dashboard/profile. There is no way to dismiss the dashboard tour without triggering the chain; only the WelcomeModal's 'O'zim ko'raman' (skipWelcome) actually opts out.
- **Fix:** Distinguish finish from abort. Give Tour.tsx separate onFinish and onClose callbacks (or pass a completed boolean to onClose), and only call startOnboarding from the genuine completion path. Wire the X/skip buttons to a plain dismiss that marks dashboard_v1 seen without starting the chain.
- **Verified:** high — Confirmed against the actual code. In frontend/src/app/dashboard/page.tsx the dashboard Tour is rendered with onClose={finishDashboardTour} (line 500), and finishDashboardTour() (lines 161-171) unconditionally calls markTourSeen then startOnboarding((p)=>router.push(p)) after a 250ms timeout — with no parameter distinguishing finish from abort. In frontend/src/components/yz/Tour.tsx the single onClose prop is invoked by every exit affordance: the X / "Yopish" button (lines 197-198), the "O'tkazib yuborish" skip button (lines 207-209), the "Tayyor" finish button on the last info step (line 216), and the action-mode completion path (line 89). Both the X and skip buttons are rendered for every step regardless of mode, so any deliberate mid-tour abort routes through onClose -> finishDashboardTour -> startOnboarding. startOnboarding (frontend/src/lib/onboarding.ts:91-102) wipes tour-seen flags, sets the active flag, and navigates to ONBOARDING_SEQUENCE[0] which is /dashboard/profile; the sequence (lines 38-46) has exactly 7 entries (profile, services, staff, promo, qr, clients, bookings). The only true opt-out is the WelcomeModal's "O'zim ko'raman" -> skipWelcome (page.tsx:153-159), which fires before the Tour mounts; once inside the Tour there is no dismiss path that avoids the chain. The code comment at page.tsx:162-165 explicitly acknowledges the conflation ("either by finishing or X-ing out... since they chose the guided path"), confirming this is unguarded behavior rather than something handled elsewhere. Adversarial counter-arguments (user consented to the tour; action-mode steps might skip onClose; already handled) all fail because choosing the WelcomeModal's "short tour" is not consent to a 7-page chain, the abort buttons are mode-independent, and no alternate dismiss path exists. The proposed fix (separate onFinish/onClose or a completed boolean, wiring X/skip to a plain dismiss) is correct and minimal. Severity (high/bug UX) is justified: a user explicitly aborting is force-navigated away into a 7-step flow.

### 2. [HIGH/correctness] Busy-slot grid treats CANCELLED bookings as busy and ignores multi-slot duration
- **Where:** `frontend/src/components/yz/NewBookingSheet.tsx:87-92`
- **Problem:** The busy-slot set is built from EVERY booking returned for the date: `apiFetch(/api/business/me/bookings?booking_date=${date})` then `new Set(rows.map(r => r.start_time.slice(0,5)))`. The GET endpoint (backend list_bookings, bookings.py:215-233) returns bookings of ALL statuses when no status filter is passed, so a CANCELLED booking at 14:00 makes 14:00 show as busy (struck-through, undisclickable) even though the slot is actually free. Conversely, the grid only marks the exact hourly start as busy: a 14:00 booking whose service lasts 90 min does NOT disable 15:00, so the owner can pick 15:00 and the create call then fails server-side with 'Vaqt band' (the backend overlap check at bookings.py:169-183 uses real start/end ranges). So the client-side grid both over-blocks (cancelled) and under-blocks (duration overflow) relative to the server's truth.
- **Fix:** Fetch busy slots with a status filter (e.g. only PENDING/CONFIRMED) or filter rows client-side to exclude CANCELLED/COMPLETED before building the Set, and compute occupied ranges from start_time+end_time (mark every hourly slot the [start,end) interval covers) instead of only the exact start hour. Better: expose/consume a dedicated free-slots endpoint so the grid matches the server's conflict logic.
- **Verified:** high — Confirmed both halves of the claim against the actual code.

OVER-BLOCKING (cancelled treated as busy): frontend/src/components/yz/NewBookingSheet.tsx:89-90 calls GET /api/business/me/bookings?booking_date=${date} with NO status filter and builds busySlots = new Set(rows.map(r => r.start_time.slice(0,5))). The backend list_bookings (backend/app/routers/bookings.py:215-233) only applies a status filter when the `status` query param is provided (lines 231-232); with no filter it returns ALL statuses including CANCELLED/COMPLETED. So a cancelled 14:00 booking lands in busySlots and the grid renders 14:00 as disabled + line-through (sheet lines 264-283) even though the slot is free.

UNDER-BLOCKING (duration ignored): busySlots keys only on the exact hourly start (start_time.slice(0,5)); a 14:00 booking with a 90-min service does not add 15:00 to the set, so 15:00 stays selectable. The server conflict check (bookings.py:169-183, owner-create path used by the sheet's POST to /api/business/me/bookings) uses real interval overlap (start_time < end_time AND end_time > start_time) restricted to PENDING/CONFIRMED, and raises HTTPException(400, "Vaqt band") on overlap. apiFetch (frontend/src/lib/api.ts:86-88) throws that body and finish() (sheet 126-128) shows it as a toast. So the client lets the owner pick 15:00 and the create then fails server-side — exactly as claimed.

Fix is feasible without backend changes: BookingRead (backend/app/schemas/booking.py:47-64) already exposes status and end_time, so the frontend can exclude CANCELLED/COMPLETED and expand [start,end) across hourly slots; the type annotation at sheet line 89 currently only reads start_time. A dedicated owner free-slots endpoint mirroring the public /{slug}/slots (bookings.py:656-672, uses get_available_slots) would be more robust. Nothing in the current code mitigates this; not a false positive.

Minor nuance: the server conflict check is per-staff when staff_id is set, but the owner sheet never sends staff_id, so the per-business framing in the claim holds for this flow. Line numbers cited (87-92, 215-233, 169-183) all match.

### 3. [HIGH/correctness] Payme/Click webhooks complete subscriptions without reconciling the paid amount
- **Where:** `backend/app/routers/payments.py:426-480`
- **Problem:** payme_webhook and click_webhook locate the transaction by regex-scanning the whole JSON for any UUID (`_find_tx_id`), then call complete_transaction_and_notify(db, tx) which activates the full plan stored on `tx` — without ever comparing the amount the provider reports to `tx.amount`. Payme auth is only Basic-auth with a static merchant key (no per-request binding), and although the Click signature includes `amount`, the handler never checks that the signed amount equals `tx.amount`. A provider callback (or anyone who obtains the static Payme key) reporting an underpayment with a structurally valid request still grants the full MONTHLY/YEARLY subscription. There is also no binding that the matched UUID is the Payme `account.id` field rather than some other UUID in the payload.
- **Fix:** In both webhooks, read the provider-reported amount and reject (or do not complete) when it does not equal `tx.amount`. For Payme also extract the transaction id from the documented `params.account.id` field rather than a blind regex over the JSON. Keep the existing COMPLETED short-circuit for idempotency.
- **Verified:** high — Confirmed from the code. In backend/app/routers/payments.py, payme_webhook (lines 426-450) and click_webhook (lines 453-480) both locate the transaction via _find_tx_id (lines 377-387), which json.dumps() the whole body and returns the FIRST UUID matched by regex — no binding to Payme's documented params.account.id. Both then call complete_transaction_and_notify(db, tx) (lines 448, 478). That function (backend/app/services/payment_service.py:204-210) unconditionally sets COMPLETED and calls activate_subscription(db, tx.business_id, plan, tx.amount) using the plan/amount stored on tx at creation time. Neither webhook ever reads the provider-reported amount and compares it to tx.amount. The ONLY use of the body's amount is line 415, inside _verify_click_signature, where it merely reconstructs the MD5 signature string — there is no equality check against tx.amount. So a structurally valid Payme callback (PerformTransaction, passing static Basic-auth) or a correctly-signed Click callback (action==1) reporting an underpayment still activates the full MONTHLY/YEARLY subscription. Grepping the entire backend shows no other webhook/amount-reconciliation logic, and the test suite (backend/tests/test_payment_service.py) does not exercise an amount-mismatch path, so this is not handled elsewhere. The proposed fix (reject when provider amount != tx.amount in both handlers; for Payme read params.account.id instead of a blind regex; keep the COMPLETED short-circuit at lines 441/475 for idempotency) is correct and actionable. Severity nuance: external forgery still requires the gateway secret (Payme static key for Basic auth; click_secret_key for the MD5 signature), so this is not an unauthenticated bypass — but the missing amount reconciliation is a real, standard control gap, most concrete on the Click path where the provider actually signs the real amount and the handler ignores it.

### 4. [HIGH/correctness] Owner 'confirm' button resurrects CANCELLED / NO_SHOW bookings
- **Where:** `backend/bot/handlers/owner.py:39-43`
- **Problem:** owner_confirm only guards against re-confirming an already-CONFIRMED booking (`if booking.status == BookingStatus.CONFIRMED`). It sets `booking.status = BookingStatus.CONFIRMED` for any other status. The original new-booking notification message keeps its inline 'Tasdiqlash'/'Rad etish' keyboard indefinitely, and several flows cancel a booking WITHOUT touching that message: client self-cancel (my_bookings.cxl: edits only the client's message), and the nightly NO_SHOW auto-flip. So an owner tapping the stale 'Tasdiqlash' button on the original notification flips a CANCELLED or NO_SHOW booking back to CONFIRMED. It also bypasses booking_service entirely, so no slot-availability recheck runs — the slot may already be taken by a different booking, producing a silent double-book. Confirmation is also asymmetric with rejection/cancel which route through booking_service.
- **Fix:** Only allow the transition from a pending state: `if booking.status != BookingStatus.PENDING: await cb.answer('Allaqachon ko'rib chiqilgan', show_alert=True); return`. Prefer routing through booking_service.confirm_booking(db, UUID(bid), biz.id) so business rules/side-effects stay centralized, and consider re-validating slot availability before flipping to CONFIRMED.
- **Verified:** high — Verified the full causal chain in the actual code. owner.py:39-43 (owner_confirm) only guards `if booking.status == BookingStatus.CONFIRMED`; any other status is unconditionally set to CONFIRMED and committed (lines 42-43), bypassing booking_service so no slot re-validation runs. Two cancel flows leave the owner's notification keyboard live: (1) my_bookings.py:356-375 client self-cancel routes through cancel_booking (status->CANCELLED) but only edits the client's own message (line 372), never the owner notification; (2) reminders.py:74-105 flag_no_shows nightly task flips PENDING/CONFIRMED to NO_SHOW (line 100) touching no Telegram messages. The new-booking owner notification carries a persistent own_confirm:{id}/own_reject:{id} inline keyboard (inline.py:130-131, bookings.py:104-105) that neither cancel path removes. So tapping the stale Tasdiqlash button flips a CANCELLED or NO_SHOW booking back to CONFIRMED with no slot recheck (genuine silent double-book risk since _check_slot_available is never called). Asymmetry is real too: own_reject/rej: route through booking_service.cancel_booking while own_confirm writes status directly. The core proposed fix (PENDING-only guard) is correct and actionable. Caveat: routing through booking_service.confirm_booking ALONE does not fix it — that function (booking_service.py:392-401) also has no status guard and unconditionally sets CONFIRMED; the load-bearing remediation is the status guard, which the claim also specifies. The bug itself is accurately described and confirmed.

### 5. [HIGH/security] JWT access token leaked into URL query string (SSE stream, brochure, settings)
- **Where:** `frontend/src/app/dashboard/page.tsx:225-228 (also qr/page.tsx:94, the brochure iframe qr/page.tsx:197)`
- **Problem:** The access JWT is appended as `?token=<jwt>` to the EventSource URL (dashboard/page.tsx:227) and to the brochure download URL handed to Telegram WebApp.openLink (qr/page.tsx:94). Tokens in URLs land in server access logs, proxy logs, browser history, and the Referer header, and openLink hands the full URL (token included) to the external system browser / Telegram link handler. A leaked access token lets an attacker impersonate the owner until it expires. The code comments acknowledge this is intentional but it is a real credential-exposure surface for a multi-tenant app.
- **Fix:** Prefer short-lived, single-purpose tokens for query-param contexts (a one-time download token minted by POST and exchanged), or move SSE auth to a cookie. At minimum scope these tokens narrowly and keep their TTL very short. Do not pass the primary access JWT in URLs.
- **Verified:** high — Confirmed from code. The primary access JWT (localStorage["yozuv_access"]) is placed in URL query strings in two live paths: (1) frontend/src/app/dashboard/page.tsx:225-228 appends ?token=<jwt> to the EventSource URL for /api/business/me/notifications/stream; (2) frontend/src/app/dashboard/qr/page.tsx:94 appends ?token=<jwt> to the brochure URL and passes it to Telegram WebApp.openLink, which hands the full URL (token included) to the external system browser. getToken()/the SSE token both read the same full-privilege access JWT (api.ts:6). The backend honors it: deps.py get_owned_business_download (lines 35-58) reads the ?token= query param and resolves it via get_user_from_token/decode_token, and both the SSE endpoint (business.py:386) and brochure endpoint (files.py:91) depend on it. The token is NOT short-lived or scoped: config.py:20 sets access_token_expire_minutes=60, and utils/auth.py mints a plain access JWT with no audience/scope narrowing, so a captured token impersonates the business owner for up to an hour. This is a genuine credential-exposure surface (server/proxy access logs, browser history, Referer, external link handler) in a multi-tenant app, and the proposed fix (short-lived single-purpose download token, or cookie-based SSE auth) is appropriate. Two citation inaccuracies do not refute the core issue: the claimed file path prefix is wrong (actual path is frontend/src/app/dashboard/qr/page.tsx, not frontend/src/app/qr/page.tsx), and the claimed brochure iframe leak at qr/page.tsx:197 is a false positive — that iframe src is a blob: URL from URL.createObjectURL (line 68) and contains no token. The two real leak sites stand independently.

### 6. [HIGH/security] Owner booking-create does not scope client lookup to the business (cross-tenant client reference)
- **Where:** `backend/app/routers/bookings.py:137-139`
- **Problem:** create_owner_booking (the endpoint NewBookingSheet POSTs to at /api/business/me/bookings) validates the service with `Service.business_id == business.id` and validates staff ownership, but the client is looked up with `db.query(Client).filter(Client.id == body.client_id).first()` with NO business_id constraint. A multi-tenant owner who supplies any other tenant's client UUID gets a booking created that references a client belonging to a different business. The normal UI never does this (NewBookingSheet only lists the owner's own clients), but the API surface this frontend subsystem drives allows cross-tenant data binding / information disclosure (the foreign client then appears on this business's booking, and reminders/notifications in reminders.py resolve client by id without tenant scoping).
- **Fix:** Filter the client query by tenant: add `Client.business_id == business.id` (or whatever the ownership column is) to the lookup, mirroring the Service/Staff checks, and 404 when the client does not belong to the caller's business. Audit the public-flow client lookup similarly.
- **Verified:** high — CONFIRMED as a real, actionable issue, though the proposed fix's specific column name is wrong.

Verified facts in D:\yozuv\backend\app\routers\bookings.py:
- create_owner_booking (POST /business/me/bookings) at lines 126-134 scopes the Service lookup with Service.business_id == business.id, and lines 146-154 scope the Staff lookup with Staff.business_id == business.id.
- BUT lines 137-139 look up the client with `db.query(Client).filter(Client.id == body.client_id).first()` with NO tenant scoping at all. An owner can supply any client UUID system-wide and get a booking created against it. The identical unscoped pattern also exists in create_recurring (lines 527-529).

The claim's PRIMARY proposed fix ("add Client.business_id == business.id") is technically impossible: D:\yozuv\backend\app\models\client.py shows Client is a GLOBAL table keyed by a unique telegram_id with NO business_id column — one client row is shared across every business they book with. However, the claim hedges with "(or whatever the ownership column is)" and the issue itself is valid.

The codebase proves this is a real invariant violation, not a false positive: D:\yozuv\backend\app\routers\clients.py defines _client_in_business_or_404 (lines 125-141) whose docstring explicitly says a client is "in" a business only if they have >=1 booking there, and warns that skipping this check is "a privacy leak (you'd learn which UUIDs exist)." That guard is deliberately applied to client_detail, patch_client, block_client. The booking-create path simply forgot to apply the same guard — a clear inconsistency.

Impact is genuine: creating a booking that references a stranger's client row (a) discloses that client's first_name/last_name/phone/telegram_id on this business's bookings, (b) afterward makes _client_in_business_or_404 PASS for that client (since they now have a booking), unlocking client_detail read, birthday patch, and block, and (c) reminders.py (line 49, send_hourly_reminders) resolves the client by id with no tenant scoping and would send a Telegram reminder to the stranger's telegram_id referencing the attacker's business. Exploitability is bounded by needing a valid random UUIDv4 (not enumerable), so severity "high" is slightly generous, but the defect is real and actionable.

Correct fix: mirror the existing _client_in_business_or_404 pattern (require the client to have a prior booking with this business, or otherwise scope to the tenant) in both create_owner_booking and create_recurring, returning 404 otherwise — NOT a non-existent Client.business_id filter.

### 7. [HIGH/security] client_detail leaks any client's PII cross-tenant by UUID

- **Verified:** high — Confirmed real at D:\yozuv\backend\app\routers\clients.py:65-98.
